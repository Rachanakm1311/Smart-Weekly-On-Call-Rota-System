Solution statement *  500 words or less — copy the text below
The Smart Weekly On-Call Rota Management System solves the manual, error-prone process of managing weekly on-call schedules for IBM application support teams across multiple locations in India.

The problem: On-call assignments were tracked in spreadsheets with no automated handling of leave, shift swaps, or regional public holidays across IBM India offices (Bangalore, Hyderabad, Pune, Chennai, Kolkata, Delhi). This led to missed rotations, engineers being assigned during their regional holidays, and no reliable audit trail.

The solution: A production-ready ASP.NET Core MVC web application backed by SQL Server. It maintains a fair Rotation Queue per team, automatically assigning the next engineer in line as Primary and the following as Backup each week. The system respects approved leave and regional public holidays — automatically skipping engineers who are unavailable. If an engineer volunteers to work on their regional holiday, they submit a Holiday Work (Comp-Off) request; once approved, they remain in the rotation pool.

Admins can generate the rota manually for any team and week, or let the system run automatically every Friday at 08:00, immediately emailing the assigned engineers. Employees have a self-service portal to view their schedule, apply for leave, request shift swaps, and receive in-app and email notifications.

IBM's ICA (Intelligent Curation Assistant) is embedded on every page via a floating button. During a P1/P2 incident, the on-call engineer clicks it, pastes the error or logs into the ICA chat, and receives AI-powered resolution guidance instantly — without switching tools. The MS-COTS team agent is live; remaining teams will plug in their ICA URLs with no code changes required.

Technical statement *  500 words or less — copy the text below
The application targets .NET 10 using ASP.NET Core MVC with Entity Framework Core 10 against SQL Server. The architecture follows a repository + service layer pattern:

IRepository<T> / GenericRepository<T> — async CRUD abstraction over every EF DbSet.
RotaService — rotation assignment: queries the RotationQueue by position, builds a skip list of employees on approved leave or regional holiday (cross-referencing HolidayCalendar.Location vs Employee.Region), assigns Primary and Backup, persists to OnCallSchedule, and fires in-app + email notifications.
WeeklyRotaBackgroundService — IHostedService that sleeps until the next configured day/time (default: Friday 08:00), then calls RotaService for every active team, generating next week's (Monday–Sunday) rota automatically.
AuthService — BCrypt.Net-Next (work factor 12) with plain-text fallback for seed data; passwords are hashed on every create/edit.
NotificationService — saves an Unread row to the Notifications table and dispatches a branded HTML email via MailKit/MimeKit (Gmail SMTP, STARTTLS port 587).

The database schema has 10 tables: Roles, Teams, Employees, Applications, HolidayCalendar, LeaveRequests, OnCallSchedule, RotationQueue, ShiftSwapRequests, and Notifications. FK constraints and delete behaviours are configured via EF Fluent API. Authentication is session-based (8-hour timeout, HttpOnly cookie). The abstract BaseController provides RequireLogin() and RequireAdmin() guards across all 14 controllers. The UI uses Bootstrap 5.3, Bootstrap Icons 1.11, and DataTables 1.13 across 35+ Razor views.

IBM Bob was used throughout: generating entity models, RotaService skip logic, the background scheduler, NotificationService email templates, the full SQL seed script (MERGE-based idempotent inserts), all 14 controller scaffolds, and the 3-minute demo artefact.
